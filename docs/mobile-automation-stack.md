# PennyForge ‚Äî Mobile Automation Stack (Capacitor + Fastlane)

How PennyForge gets from **GitHub ‚Üí build ‚Üí TestFlight** with a wrapped web app,
and exactly what's already staged in this repo to make that a near-one-command
activation. This is the mobile-pipeline companion to `docs/tooling-options.md`
(the tool audit) and `docs/ios-ci-cd-options.md` (which cloud runs the build).
Distribution-path rationale (Capacitor vs Expo vs SwiftUI) lives in
`docs/ios-roadmap.md`; App Store submission requirements in
`docs/app-store-checklist.md`.

> Nothing here is active yet. The files below are **staged and CI-safe** ‚Äî they
> add no dependencies to `package.json` and don't run in CI, so the
> `build-and-test` check stays green until you deliberately activate them.

## Part 1 ‚Äî Capacitor (the iOS wrapper)

### Why a server-URL shell, not a static export

PennyForge is server-rendered: server components fetch via `lib/db.ts`, `/api`
routes run Prisma, and there is no static `out/` bundle. So the Capacitor shell
uses **`server.url`** pointing at the hosted deployment (`capacitor.config.ts`),
matching the "native shell pointed at the hosted Next.js app" model in
`docs/ios-roadmap.md`. This keeps ~90%+ of the code shared and means a web deploy
ships to iOS simultaneously.

The tradeoff: the app needs network to function (no offline mode ‚Äî explicitly
deferred per `docs/product-spec.md`), and App Review will scrutinize a webview
shell under Guideline 4.2 (see "App Store review" below).

### Packages installed on activation

`scripts/ios-bootstrap.sh` installs these (not in `package.json` yet, to keep the
lockfile in sync and CI green):

| Package | Purpose |
|---|---|
| `@capacitor/core`, `@capacitor/cli` | Capacitor runtime + CLI |
| `@capacitor/ios` | The iOS platform |
| `@capacitor/camera` | Receipt/shelf-tag photo capture (the key 4.2 de-risker) |
| `@capacitor/geolocation` | "Distance from home" for the route planner |

Push (`@capacitor/push-notifications`) is intentionally **not** included ‚Äî it's
additive later at the `lib/alerts.ts` swap point (`docs/ios-roadmap.md`), not a v1
requirement.

### Project structure after activation

```
capacitor.config.ts        # committed now (appId, appName, server.url) ‚Äî tsconfig-excluded
ios/                        # generated by `npx cap add ios` (gitignore build artifacts, commit the project)
  App/
    App.xcworkspace         # what you open in Xcode / what CI builds
    App.xcodeproj
    App/Info.plist          # add the permission strings below
    fastlane/               # copied from tooling/ios/fastlane by the bootstrap script
    Gemfile                 # pins fastlane
    ExportOptions.plist
```

### Xcode handoff (one-time, on a Mac)

1. `npm run ios:bootstrap` ‚Äî installs deps, runs `npx cap add ios`, copies the
   Fastlane templates in.
2. `npm run cap:open` ‚Äî opens `App.xcworkspace` in Xcode.
3. In Xcode: set **Signing ‚Üí Team**, confirm the **bundle ID** `com.pennyforge.app`,
   set the display name, drop in the app icon (1024¬≤ master ‚Äî `docs/app-store-checklist.md`
   ¬ß9).
4. Add the **Info.plist permission strings** (below), archive (**Product ‚Üí
   Archive**), and do one manual TestFlight upload to prove the path before
   automating it.

### Native permission strings (Info.plist)

Only declare a permission once the corresponding feature is wired ‚Äî an unused
permission is an App Review rejection (`docs/app-store-checklist.md` ¬ß5‚Äì6). Copy is
reproduced there; the near-term set:

```xml
<key>NSCameraUsageDescription</key>
<string>PennyForge uses your camera to photograph receipts and shelf tags as proof
for your clearance and penny-item reports, and to scan barcodes for quick lookup.</string>

<key>NSLocationWhenInUseUsageDescription</key>
<string>PennyForge uses your location to show nearby stores with clearance and
penny-item leads and to plan efficient routes. Your location is never shown to
other users.</string>
```

Use **When In Use** location only (no background use case in the MVP). Add
`NSPhotoLibraryUsageDescription` only if/when library attach ships.

### Build implications

- **A hosted backend is mandatory** before any real build ‚Äî the shell can't point
  at local SQLite (`docs/app-store-checklist.md` ¬ß11). Vercel + Supabase first.
- `npx cap sync ios` must run **after** `npm run build` in CI so the shell picks up
  the latest config/assets. Both staged runners do this.
- Server-URL mode means most "app updates" are just web deploys ‚Äî you only submit a
  new binary when native code/permissions/plugins change. That's a big release-
  cadence win.

### npm scripts added

```jsonc
"ios:bootstrap": "bash scripts/ios-bootstrap.sh",  // one-time activation
"cap:sync":  "cap sync ios",                        // rebuild + copy web ‚Üí native
"cap:copy":  "cap copy ios",                        // copy web assets only
"cap:open":  "cap open ios",                         // open Xcode workspace
"ios:beta":    "cd ios/App && bundle exec fastlane beta",
"ios:release": "cd ios/App && bundle exec fastlane release"
```

`cap`/`bundle` resolve only after `ios:bootstrap` installs them ‚Äî that's expected.

## Part 2 ‚Äî Fastlane (the release automation)

Fastlane turns "build + sign + upload" into `fastlane beta`, run identically on a
laptop or any CI runner ‚Äî so the release logic isn't locked to one CI vendor.

### Files (staged in `tooling/ios/`, copied to `ios/App/` on bootstrap)

| File | Role |
|---|---|
| `fastlane/Fastfile` | `beta` (‚Üí TestFlight) and `release` (‚Üí App Store) lanes |
| `fastlane/Appfile` | App identifier + Apple/team IDs, all read from env |
| `fastlane/Matchfile` | Optional `match` code-signing config |
| `Gemfile` | Pins `fastlane` + `cocoapods` for reproducible runs |
| `ExportOptions.plist` | App Store export options for `gym`/`xcodebuild` |

### What the lanes do

- **`beta`** ‚Äî resolves the App Store Connect API key ‚Üí (optional `match`) ‚Üí
  bumps the build number to `latest_testflight_build_number + 1` ‚Üí `build_app`
  (app-store export) ‚Üí `upload_to_testflight`. This is the GitHub‚ÜíTestFlight path.
- **`release`** ‚Äî same build, then `upload_to_app_store` with
  `submit_for_review: false` so you eyeball the build in App Store Connect before
  releasing. Flip to `true` once trusted.

### Auth: App Store Connect API key (preferred)

The Fastfile uses an **API key**, not Apple-ID login, because it works in CI
without an interactive 2FA prompt. Create it at App Store Connect ‚Üí Users and
Access ‚Üí Integrations ‚Üí Keys, download the `.p8` once, and record the Key ID +
Issuer ID.

### Required secrets

Store these in your CI's secret store (GitHub Actions secrets *or* a Codemagic env
group ‚Äî see `docs/ios-ci-cd-options.md`), never in git:

| Env var | What |
|---|---|
| `ASC_KEY_ID` | API key ID |
| `ASC_ISSUER_ID` | API issuer ID |
| `ASC_KEY_P8` | **base64** of the `.p8` contents |
| `DEVELOPER_TEAM_ID` | 10-char Apple team ID |
| `FASTLANE_APPLE_ID` | Apple ID email (fallback auth) |
| `MATCH_GIT_URL` | *optional* ‚Äî private certs repo (only if using `match`) |
| `MATCH_PASSWORD` | *optional* ‚Äî encrypts the match repo |

To produce `ASC_KEY_P8`: `base64 -i AuthKey_XXXXXX.p8 | pbcopy` and paste the
result as the secret value.

### Signing: automatic vs match

- **Automatic (recommended to start):** leave `MATCH_GIT_URL` unset; the Fastfile
  skips `match` and relies on Xcode automatic signing + `ExportOptions.plist`.
  Simplest for a solo dev.
- **`match` (scale later):** set `MATCH_GIT_URL`/`MATCH_PASSWORD` to share certs +
  profiles across machines/runners via a private git repo. Better once there's more
  than one build machine, but adds a repo to maintain.

## Part 3 ‚Äî activation checklist (do this when iOS work starts)

1. [ ] Vercel is live and you have the production URL.
2. [ ] Apple Developer account active; bundle ID `com.pennyforge.app` registered;
       App Store Connect app record created (`docs/app-store-checklist.md` ¬ß1‚Äì3).
3. [ ] App Store Connect API key created; `.p8` + Key ID + Issuer ID saved.
4. [ ] On a Mac: `export CAPACITOR_SERVER_URL=https://<your-vercel-url>` then
       `npm run ios:bootstrap`.
5. [ ] `npm run cap:open`, set Team + bundle ID, add Info.plist strings + icon.
6. [ ] Manual **Product ‚Üí Archive** ‚Üí upload once, to prove the path end-to-end.
7. [ ] Commit the generated `ios/` project (build artifacts gitignored).
8. [ ] Add the Fastlane secrets to your chosen CI runner.
9. [ ] `cd ios/App && bundle install && bundle exec fastlane beta` locally to
       confirm the lane before wiring CI.
10. [ ] Turn on the runner (`docs/ios-ci-cd-options.md`) ‚Äî Codemagic (`ios-v*`
        tag) or GitHub Actions (`workflow_dispatch`).

## What is NOT in scope here (deferred by design)

- **Push notifications (APNs)** ‚Äî additive later via `@capacitor/push-notifications`
  at the `lib/alerts.ts` swap point; the in-app alert inbox works in the shell today.
- **Offline mode** ‚Äî the server-URL shell needs network; offline queueing is
  deferred (`docs/product-spec.md`).
- **Receipt OCR / barcode scan** ‚Äî camera capture ships first; OCR and scanning are
  Phase 2/3.
- **Android** ‚Äî Capacitor supports it, but there's no Android target in scope now.
